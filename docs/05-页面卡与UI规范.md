## 页面卡 1：用户首页

| 项目 | 内容 |
|------|------|
| **页面名称** | 用户首页 |
| **页面角色** | 系统入口 + 身份切换 |
| **页面目标** | 用户打开网站第一眼看到这个页面，在这里选择体验账号，然后导航到功能页面 |
| **页面入口** | 直接访问 `/` |
| **页面出口** | 点击导航栏"座位列表"→ 座位列表页；点击"我的预约"→ 我的预约页 |
| **最重要的信息** | 当前登录用户（"当前用户：学生A"）+ 系统名称 |
| **最重要的操作** | 体验账号下拉切换 |
| **成功状态** | 页面正常加载，顶部下拉框可选中账号，选中后页面刷新显示该用户名 |
| **空状态** | 本项目不涉及（预设 5 个体验账号固定存在） |
| **异常提示** | 无（首页没有后端请求） |
| **手机端/桌面端** | 手机优先：顶部导航栏+账号切换占满宽度；桌面居中有最大宽度 960px |
| **后端关键点** | 体验账号为硬编码种子数据（StudentUser 表或内存列表），不涉及 API 请求；切换后写入 Session |

## 页面卡 2：座位列表页

| 项目 | 内容 |
|------|------|
| **页面名称** | 座位列表页 |
| **页面角色** | 浏览所有座位 + 按条件筛选 |
| **页面目标** | 用户可以快速看到哪些座位空闲，按区域/楼层缩小范围 |
| **页面入口** | 首页 → 点击"座位列表"（路由 `/Seats`） |
| **页面出口** | 点击座位卡片 → 座位详情页；导航栏→ 首页 / 我的预约 |
| **最重要的信息** | 每个座位的状态（空闲/已预约/使用中/维护中） |
| **最重要的操作** | 筛选下拉框 + 点击座位卡片 |
| **成功状态** | 页面显示 ≥10 个座位卡片，每个卡片用颜色标识状态 |
| **空状态** | 该区域无座位 → 显示"该区域暂无座位" |
| **异常提示** | 后端查询失败 → 显示"加载失败，请稍后重试" |
| **手机端/桌面端** | 手机竖屏：卡片单列或双列，筛选下拉框在上方固定；桌面：卡片三列或四列网格 |
| **后端关键点** | 查询所有座位 + 当前预约数据（用于动态计算状态）；按 Area 和 Floor 筛选；支持两个 GET 参数 `?area=xxx&floor=xxx` |

## 页面卡 3：座位详情页

| 项目 | 内容 |
|------|------|
| **页面名称** | 座位详情页 |
| **页面角色** | 展示单个座位的完整信息 + 今日预约时段预览 |
| **页面目标** | 让用户决定是否预约这个座位 |

## 页面卡 4：预约提交页

| 项目 | 内容 |
|------|------|
| **页面名称** | 预约提交页 |
| **页面角色** | 选择时间段 → 提交预约 |
| **页面目标** | 用户选择开始时间和结束时间，确认后提交预约 |
| **页面入口** | 座位详情页 → 点击"预约此座位"（路由 `/Reservation/Create/{seatId}`） |
| **页面出口** | 提交成功 → 跳转我的预约页；提交失败 → 当前页显示错误；点击"返回"→ 座位详情页 |
| **最重要的信息** | 座位号（只读）+ 开始时间 + 结束时间 |
| **最重要的操作** | 提交按钮 |
| **成功状态** | 表单校验通过 → 跳转我的预约页并显示"预约成功"消息 |
| **空状态** | 不涉及（表单页） |
| **异常提示** | 选时冲突 → 当前页红字提示"该时段已被预约"；时间规则不符 → 提示"最短 30 分钟，最长 4 小时"；未登录 → 提示"请先选择体验账号" |

| **后端关键点** | POST `/Reservation/Create` 接收 `seatId, startTime, endTime`；后端做冲突校验 + 时间规则校验；校验通过后写入 Reservation 表，状态为"待开始" |

## 3. 视觉方向说明

### 主方向：**安静学习感 + 校园服务感**

为什么是这个方向：
- 目标用户是大学生，使用场景是图书馆（安静环境）
- 不需要花哨的动效和夸张的视觉冲击
- 需要给人可靠、干净、不分散注意力的感觉
- 管理端则应偏向"现代后台感"——清晰、高效、信息密度高

### 设计关键词
- 干净留白、卡片化、柔和圆角、低阴影、高可读性

### 禁用风格
- ❌ 暗黑/深色模式（本期不做）
- ❌ 玻璃拟态（毛玻璃效果）
- ❌ 渐变背景或大幅插图
- ❌ 动效/转场动画
- ❌ 自定义字体图标（使用 Bootstrap Icons 即可）
- ❌ 大面积品牌色（主色仅用于导航和操作按钮）

## 6. 响应式原则

| 断点 | 范围 | 布局策略 |
|------|------|----------|
| 手机 | < 768px | 单列，卡片全宽，表格横滚或简化 |
| 平板 | 768px ~ 1024px | 双列，卡片宽度自适应 |
| 桌面 | > 1024px | 多列（3~4列座位卡片），最大内容宽度 1200px 居中 |

具体规则：
- 用户端所有页面优先设计 **375px 宽度（iPhone SE 宽度）** 的布局，再扩展到桌面。
- 管理端所有页面优先设计 **1280px 宽度** 的布局，再收窄到手机。
- Bootstrap 的 `container` + `row` + `col-*` 网格系统即可满足需求。
- 不需要写自定义媒体查询，Bootstrap 内置断点足够。

## 8. 哪些页面先做低保真，哪些页面直接做中保真

### 先做低保真（全部 9 页）
快速摆出每个页面的区块位置，不需要漂亮，只需要确认"东西放对了"。

| 检查要点 |
|----------|
| 信息展示区够不够大 |
| 操作按钮是否在视线自然落点 |
| 手机屏上内容是否太挤 |

### 直接做中保真的 4 页（核心 + 高风险）

| 页面 | 原因 |
|------|------|
| 座位列表页 | 卡片布局 + 筛选 + 状态颜色，布局最复杂 |
| 预约提交页 | 表单字段多 + 时间选择 + 校验反馈，交互最密集 |
| 我的预约页 | 列表 + 状态标签 + 取消按钮共存，状态最多 |
| 座位管理页 | 表格 + 添加表单共存，管理端核心操作页 |

## 9. 原型评审时要重点看什么

| 序号 | 评审重点 | 说明 |
|------|----------|------|
| 1 | **页面数量是否和页面树一致** | 9 页一张不能少，也不能多 |
| 2 | **每个页面的出口是否画对** | 按钮/链接是否指向正确的下一个页面 |
| 3 | **手机竖屏下内容不溢出** | 以 375px 宽度检查每个用户端页面 |
| 4 | **状态颜色是否可区分** | 4 种座位状态 + 4 种预约状态的颜色是否色盲友好（推荐用文字+颜色双重标识） |
| 5 | **空状态是否覆盖** | 我的预约（无记录）、座位列表（筛选无结果）、管理端表格（无数据） |
| 6 | **错误反馈位置是否明显** | 表单校验错误是否出现在对应字段旁边或上方 |
| 7 | **导航路径是否闭环** | 用户是否能从任意页面回到首页或主要页面 |
| 8 | **管理端守卫是否画出** | 未登录时访问管理端是否画出跳转逻辑（可在路由说明中标注） |
