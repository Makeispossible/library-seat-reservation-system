# 需求分析与 MVP 确认 — 图书馆座位预约系统

来源：`docs/01-项目立项单.md`
制定方式：课堂替代调研（基于校园场景假设 + 同类系统参考 + 课堂工作量评估）

---

## 1. 目标用户与核心场景

### 用户角色

| 角色 | 身份 | 技术能力 | 使用方式 |
|------|------|----------|----------|
| **学生（主要用户）** | 在校大学生 | 会用浏览器 | 手机 / 电脑访问，无需注册 |
| **管理员** | 图书馆工作人员 | 电脑基础操作 | 电脑访问，账号密码登录 |

### 核心场景

**场景一：高峰期找座位**
> 期末周图书馆爆满，学生 A 在走廊转了两圈找不到座位。他用手机打开预约系统，看到 3 楼还有 2 个空闲座位，直接预约了一个 2 小时的时段，到座位坐下。

**场景二：预约后不去了**
> 学生 B 预约了一个下午的座位，但临时有事去不了。他在手机上打开"我的预约"，点击取消，座位状态立即释放，可以被其他同学预约。

**场景三：管理员日常维护**
> 图书馆老师在后台看到某张桌子的椅子坏了，进入座位管理将该座位状态设为"维护中"。前台列表中该座位不再显示为可选。

**场景四：周一早上了解上座情况**
> 图书馆老师登录后台查看统计页，看到上周三四楼座位使用率最高，决定调整各楼层的桌椅配置。

## 2. 用户最关心的问题

### 学生关心的

1. **"还有没有空座？"**—— 能不能快速看到哪些座位是空闲的。
2. **"怎么预约？"**—— 操作是不是简单，点几下就能约好。
3. **"约了不去怎么办？"**—— 能不能取消，取消麻不麻烦。
4. **"要注册账号吗？"**—— 不想填手机号、验证码，最好点一下就能用。
5. **"手机上看会不会乱？"**—— 能不能在手机浏览器上正常操作。

### 管理员关心的

1. **"怎么加一个座位？"**—— 新买了桌椅能不能自己添加上去。
2. **"谁约了哪个座位？"**—— 能不能看到所有预约记录。
3. **"哪些座位最受欢迎？"**—— 有没有数据帮我决定怎么分配座位。

## 5. MVP 范围确认

### MVP 包含功能（P0 + P1）

**学生端：**

- 体验账号切换 + 当前用户显示 + Session 保持
- 座位列表页（显示所有座位+状态）
- 座位详情页（含今日预约时段预览）
- 按区域/楼层筛选
- 选择时间段提交预约
- 预约冲突校验 + 时间段合理性校验
- 预约成功/失败反馈提示
- 我的预约列表
- 取消预约（含确认弹窗）
- 种子数据预置

**管理端：**

- 管理员登录
- 座位列表
- 添加座位
- 查看全部预约记录

**公共：**

- EF Core 数据库迁移
- Bootstrap 响应式（优先 PC，有余力再做手机适配）

### MVP 共约 **9 个页面**

| # | 页面 | 层级 |
|---|------|------|
| 1 | 用户首页—模拟登录 | P0 |
| 2 | 座位列表 | P0 |
| 3 | 座位详情 | P1 |
| 4 | 预约提交 | P0 |
| 5 | 我的预约 | P0 |
| 6 | 管理员登录 | P0 |
| 7 | 管理端-座位列表+添加 | P1 |
| 8 | 管理端-预约记录 | P0 |
| 9 | 操作结果提示页 | P1 |

删除的页面：取消预约（合并到"我的预约"）、座位编辑/删除管理（降级）、统计页（暂缓）

## 6. 本期明确不做的功能

### 从立项单继承的

- 支付功能
- 扫码签到
- 消息推送
- 复杂账号体系
- 多馆区联动
- 违规扣分/信用分
- 座位热力图/三维地图
- 移动端 App

### 本阶段从候选功能池中裁剪的

| 功能 | 裁剪原因 |
|------|----------|
| 统计页（C9） | 数据量太少做统计没意义，演示时用数据库查询截图替代 |
| 编辑/删除座位（C4、C5） | MVP 只做"添加"，编辑和删除用数据库工具处理 |
| 手动结束预约（C8） | 演示时可以改数据库状态，无需专门页面 |
| 404/500 错误页 | ASP.NET Core 自带默认错误页，够用 |
| 空状态提示 | 种子数据保证运行后不空，不独立开发 |

## 8. 进入 PRD Lite 前的最终口径

### 一句话项目定位

> 一个只做核心预约流程的图书馆座位预约系统，学生选账号即用、查座即约、约了能取消；管理员能加座位、看到所有预约。

### 需要 PRD Lite 中明确的设计决策

1. **模拟登录方式**：页面顶部下拉框选择预设账号。
2. **座位筛选维度**：区域 + 楼层（两级筛选）。
3. **预约时间段粒度**：最小 30 分钟，最长 4 小时，不可跨天。
4. **预约冲突规则**：同一座位同一时间区间只能有一个有效预约。
5. **座位状态定义**：空闲 / 已预约 / 使用中 / 维护中。
6. **预约状态定义**：待开始 / 使用中 / 已完成 / 已取消。
7. **管理员账号**：种子数据预置一个固定账号（admin / admin123）。
8. **数据库 LocalDB 还是 SQLite**：优先 LocalDB，如果环境不支持则降级 SQLite。

### 剩余工时预估

- P0 + P1 共 17 个功能，预计 9 个页面
- 参考立项单里程碑：第 1 天环境搭建，第 2~3 天学生端，第 4 天管理端
- MVP 裁剪后，第 4 天管理端工作量明显减轻，可预留半天做打磨和测试
