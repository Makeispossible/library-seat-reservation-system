## 1. 系统设计目标

| 目标 | 说明 |
|------|------|
| 可用性优先 | 项目可通过 `dotnet run` 启动，浏览器打开即用，无需额外配置 |
| 学生无注册 | 使用体验账号切换（学生A~E），Session 持久化，不建账号体系 |
| 轻量管理 | 管理端 Session 认证，不引入 Identity Framework |
| 编码量可控 | 严格控制页面数量（9 页），不做本期范围外的功能 |
| 课程可交付 | 源码结构清晰，注释完整，适合作为课程作业提交 |

3. 连接字符串改为 `"Data Source=LibrarySeat.db"`

### 为什么不引入 Service 层？

本项目逻辑集中在"查询 + 动态状态计算"，复杂度不足以单独抽 Service 层。所有数据操作直接通过 Controller 调用 `AppDbContext`，动态状态计算通过 `StatusHelper` 静态方法完成。规则：

- 数据查询 → Controller 中用 LINQ 直接查询
- 动态计算 → `StatusHelper` 静态方法
- 认证校验 → `AdminAuthFilter` ActionFilter
- 超过 50 行的数据查询逻辑 → 提取到 `Helpers/` 下的私有方法

## 4. 分层职责说明

### Controller 层（Controllers/）

| Controller | 路由 | 职责 |
|------------|------|------|
| `HomeController` | `/`, `/Home/SwitchUser` | 渲染首页，处理账号切换 POST |
| `SeatsController` | `/Seats`, `/Seats/Detail/{id}` | 座位列表（含筛选）、座位详情 |
| `ReservationController` | `/Reservation/Create`, `/Reservation/My`, `/Reservation/Cancel` | 提交预约（GET 表单 + POST 提交）、我的预约列表、取消预约 POST |
| `AdminController` | `/Admin/Login`, `/Admin/Logout`, `/Admin/Reservations`, `/Admin/Seats`, `/Admin/CreateSeat`, `/Admin/Stats` | 管理端全部功能 |

**Controller 编码原则：**
- Action 方法不超过 30 行，超过则抽取私有方法
- 通过构造函数注入 `AppDbContext`
- 不直接操作 `HttpContext.Session` 以外的 HTTP 底层对象
- 所有 POST 操作后使用 `RedirectToAction` + `TempData`，避免重复提交

### 数据访问层（Data/）

| 文件 | 职责 |
|------|------|
| `AppDbContext.cs` | 定义 DbSet，配置实体映射（表名、主键、外键、字段约束） |
| `DbInitializer.cs` | 系统启动时执行，检测数据库是否已初始化，写入种子数据 |

**数据访问原则：**
- 查询逻辑直接在 Controller 中写 LINQ，不额外封装 Repository
- 写入操作用 EF Core 原生 `Add`/`SaveChanges`
- 所有查询使用 `IQueryable` 延迟执行，避免提前 `.ToList()` 导致全表加载

### 视图模型层（Models/ 中的非实体类）

| 类 | 用途 |
|----|------|
| `SeatDisplay` | 座位列表页使用，包含 `Id/SeatNumber/Area/Floor/DisplayStatus` 五个字段 |
| ViewBag | 除 `SeatDisplay` 外的零散数据通过 `ViewBag` 传递（筛选选项列表、当前用户等） |

**原则：** 坚持简单的数据传递方式。视图模型只在"需要在多个地方复用的数据结构"时才创建。一次性的展示数据直接通过 `ViewBag` 或 `dynamic`。

### 辅助层（Helpers/）

| 文件 | 职责 |
|------|------|
| `StatusHelper.cs` | 两个静态方法：`GetSeatDisplayStatus`（计算座位可见状态）、`GetReservationDisplayStatus`（计算预约记录可见状态） |

### 过滤器（Filters/）

| 文件 | 职责 |
|------|------|
| `AdminAuthFilter.cs` | ActionFilter 属性，标注在需要登录的 Admin Action 上。检查 Session 中 `AdminLoggedIn` 是否存在，不存在则重定向到 `/Admin/Login` |

## 7. 主链路工程落点

### 链路 A：完整预约流程（5 步）

```
步骤          | 工程落点
① 用户首页    | HomeController.Index() → 渲染 Views/Home/Index.cshtml
② 座位列表    | SeatsController.Index() → StatusHelper.GetSeatDisplayStatus()→ Views/Seats/Index.cshtml
③ 座位详情    | SeatsController.Detail() → Views/Seats/Detail.cshtml
④ 提交预约    | ReservationController.Create(POST) → 冲突校验 → 写入Reservation → 重定向
⑤ 我的预约    | ReservationController.My() → Views/Reservation/My.cshtml
```

### 链路 B：取消预约流程（3 步）

```
步骤          | 工程落点
① 我的预约页  | ReservationController.My() → 渲染列表 + 取消按钮
② 确认弹窗    | Bootstrap Modal（前端，无需后端）
③ 执行取消    | ReservationController.Cancel(POST) → Status → "Cancelled" → 重定向My
```

### 链路 D：管理登录 + 管理座位（4 步）

```
步骤          | 工程落点
① 管理登录    | AdminController.Login(POST) → Session写入
② 预约管理    | AdminController.Reservations() → [AdminAuth] → Views/Admin/Reservations.cshtml
③ 添加座位    | AdminController.CreateSeat(POST) → 唯一性校验 → 写入Seat
④ 前台验证    | SeatsController.Index() → 新座位在列表中（无需额外代码）
```

## 8. 异常处理原则

| 异常场景 | 处理方式 | 示例 |
|----------|----------|------|
| 数据库查询失败 | ASP.NET Core 默认错误页（开发模式显示异常详情） | 无特殊处理，由框架兜底 |
| 参数校验失败 | 服务端 ModelState 校验 + 当前页显示错误消息 | 时间规则不符 → 表单上方红字提示 |
| 业务冲突 | Controller 中显式检查 + 返回当前页 + 错误消息 | 时段冲突 → "该时段已被预约" |
| 资源不存在 | `return NotFound()` → ASP.NET Core 404 | 座位 ID 无效 |
| 越权操作 | 检查 Session → 不匹配则 TempData 错误 + 重定向 | 取消别人的预约 |
| 管理端未登录 | `AdminAuthFilter` → 自动重定向到 `/Admin/Login` | 直接访问 `/Admin/Seats` |

### 不使用 try-catch 的页面

所有 Controller Action 不包裹 try-catch。EF Core 异常会由 ASP.NET Core 中间件统一处理，开发环境显示黄页，生产环境显示默认错误页。本项目运行在开发环境，不需要自定义异常处理。

### TempData 消息

所有操作结果（成功/失败）通过 `TempData` 传递到下一个请求：

```csharp
TempData["SuccessMessage"] = "预约成功！";
TempData["ErrorMessage"] = "该座位编号已存在";
```

在 Layout 中统一渲染：

```html
@if (TempData["SuccessMessage"] != null) { <div class="alert alert-success">@TempData["SuccessMessage"]</div> }
@if (TempData["ErrorMessage"] != null) { <div class="alert alert-danger">@TempData["ErrorMessage"]</div> }
```

## 9. Session / Cookie 使用原则

| 用途 | Key | 类型 | 生命周期 | 说明 |
|------|-----|------|----------|------|
| 当前体验账号 | `UserId` | `int` | 30 分钟无操作过期 | 用户在首页切换账号时写入 |
| 当前体验账号名 | `UserName` | `string` | 同上 | 仅用于 ViewBag 展示 |
| 管理端登录标志 | `AdminLoggedIn` | `string` | 30 分钟无操作过期 | 登录成功写入"true" |
| 管理端用户名 | `AdminName` | `string` | 同上 | 展示用 |

### 配置（Program.cs）

```csharp
builder.Services.AddDistributedMemoryCache();
builder.Services.AddSession(options =>
{
    options.IdleTimeout = TimeSpan.FromMinutes(30);
    options.Cookie.HttpOnly = true;
    options.Cookie.IsEssential = true;
});
app.UseSession();
```

### 规则

- Session 数据不持久化到数据库，重启即丢失
- 体验账号切换仅在内存中，不涉及 Cookie 除 Session ID 外的写入
- 管理端登出时从 Session 中移除 `AdminLoggedIn`
- 所有管理端 Action 通过 `[AdminAuth]` 过滤器检查 Session，不手动检查

## 10. 当前阶段实现边界

### 本期实现

| 模块 | 边界 |
|------|------|
| 体验账号 | 5 个硬编码种子学生，通过 Session 切换，无注册 |
| 座位列表 | 12 个种子座位，3 个区域，按区域/楼层筛选 |
| 预约 | 仅限当天预约，时间范围 08:00~22:00，30 分钟粒度 |
| 预约规则 | 最短 30 分钟，最长 4 小时，不可跨天 |
| 冲突检测 | 同一座位同一时间段不能有重叠的"待开始"预约 |
| 管理端认证 | 固定账号 admin/admin123，Session 存储 |
| 座位管理 | 仅添加，不支持编辑/删除（可用数据库工具修改） |
| 统计 | 纯展示，数字由后端实时计算 |

### 本期不做

| 功能 | 原因 |
|------|------|
| 用户注册/登录 | 体验账号方案已覆盖 |
| 跨天预约 | 复杂度增加，MVP 不需要 |
| 编辑/删除座位 | 数据库工具可替代 |
| 签到/扫码 | 立项单明确不做 |
| 消息推送/通知 | 立项单明确不做 |
| 自定义错误页 | ASP.NET Core 默认页够用 |
| 高保真 Figma 原型 | 直接用 Bootstrap 进入开发 |
| JavaScript 前端校验 | 本期仅服务端校验，JS 不做（Bootstrap 自带校验除外） |
| 单元测试 | 课程项目不做测试覆盖要求 |

### 架构限制说明

- 不使用 Areas 功能：管理端通过路由前缀 `/Admin` 区分，控制器独立
- 不使用 Identity Framework：管理员认证使用 Session + 硬编码校验
- 不使用 Repository 模式：EF Core 的 DbSet 本身已是 Repository
- 不使用 AutoMapper：实体和视图模型结构简单，手动赋值
- 不使用前端构建工具：Bootstrap 通过 CDN 引入，无 Webpack/Vite
