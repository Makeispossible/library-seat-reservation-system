# 18 — 项目升级路线与展示级打磨规划

> 本文档为《图书馆座位预约系统》完成交付后的升级路线规划。
> 目标：在不重写体系结构的前提下，用有限投入让项目从"课堂作业"升级为"面试可用产品原型"。

---

## 1. 当前项目真实基线判断

### 1.1 已实现内容

| 模块 | 当前状态 | 实现深度 |
|------|---------|---------|
| 用户端选座预约 | 完整实现 | 列表 → 详情 → 提交 → 我的预约 → 取消，全链路 |
| 管理端后台 | 完整实现 | 登录 → 预约管理 → 座位管理 → 统计页 → 登出 |
| 数据库 | Code First + 种子数据 | 3 表 + 4 索引 + 双外键 + 自动迁移 |
| 静态原型 | 9 页 HTML | 用户端 5 页 + 管理端 4 页，Bootstrap 5.3 |
| 自动化测试 | Playwright 9/9 + 烟雾 10/10 | 覆盖用户端 4 用例 + 管理端 5 用例 |
| 文档 | 17 阶段全链路 | 从立项到交付，每阶段含审计跟踪 |

### 1.2 还缺什么（按差距从大到小排列）

| 差距维度 | 当前问题 | 影响程度 |
|---------|---------|---------|
| 视觉精致度 | Bootstrap 默认风格，无品牌设计系统 | 第一眼印象是"课堂作业" |
| 交互细节 | 无加载状态、无过渡动效、无实时反馈 | 操作感弱 |
| 前端工程化 | 无打包工具、无 CSS 预处理、无 TypeScript | 与现代前端实践脱节 |
| 异常路径覆盖 | 已实现基本校验，但缺 4xx/5xx 友好页面 | 出错时体验断裂 |
| 数据纵深 | 仅 12 个种子座位，无长时间数据积累场景 | 统计页展示单薄 |
| 部署 | 仅 dotnet run，无 Docker/npm build 生产流程 | 难以快速展示给面试官 |
| 安全 | 仅基础 Session 守卫，无防重放/防 CSRF/XSS 纵深 | 真实部署有风险 |
| 配置 | 连接字符串在 appsettings.json 明文 | 不适合公开仓库 |

### 1.3 最像课堂项目的地方

1. **Bootstrap 默认外观** — 无自定义设计系统、无品牌色、无字体体系，一眼即识"课程作业"
2. **统计页数据单薄** — 12 个座位 + 少量预约记录，图表区域空旷
3. **URL 路径原始** — `/Seats/Detail/1`、`/Admin/Login` 无 SEO 无路由美化
4. **无错误页面** — 404/500 没有自定义页面

### 1.4 最有潜力继续拔高的地方

1. **架构底子好** — ASP.NET Core MVC 分层清晰，Controller → Service 拆分空间大
2. **数据模型干净** — 3 表简单可扩展，适合加功能不加复杂度
3. **已有测试基线** — Playwright 9/9 通过，后续加功能不易回归
4. **文档体系完整** — 17 阶段跟踪，面试时能展示工程过程而不只是结果

## 3. 最值得优先投入的升级项清单

### 升级项 1：视觉设计系统重构

**当前问题表现：**
- 使用 Bootstrap 5.3 默认蓝色，无品牌色
- 字体为系统默认（无中文字体栈）
- 卡片间距、阴影、圆角无统一 Token
- 导航栏无品牌标识 Logo 区

**为什么会拉低项目成熟度：**
招聘方看仓库的第一件事是截图或打开首页。Bootstrap 默认外观 = 立即判定为"课堂作业"，后续功能再完整也难以扭转第一印象。

**对面试展示的价值：**
一个有设计感的首页能在 3 秒内建立"这个人有产品意识"的印象，比任何 README 描述都有效。

**对接近商业化的价值：**
设计系统是商业产品和课堂作业最直观的分界线。

**AI 可参与的工作点：**
- 让 AI 生成品牌色方案（主色/辅色/功能色/中性色 Token）
- AI 提供 CSS Variables 体系重构方案
- AI 生成 Logo（SVG 文字标或图标标）
- AI 提供中文字体栈建议（如系统字体栈）

**建议开发步骤：**
1. 定义品牌色彩系统（主色 `#1A73E8` 或自定义 + 8 种功能色 Token）
2. 在 `wwwroot/css/` 中创建 `design-tokens.css`（CSS 自定义属性体系）
3. 重构 `style.css` 引用 Token 而非字面值
4. 设计导航栏 Logo 区域（CSS 文字标即可）
5. 统一卡片、按钮、表单、表格的间距圆角体系

**关键产物：**
- `wwwroot/css/design-tokens.css` — 设计 Token 文件
- 更新后的 `wwwroot/css/style.css` — 基于 Token 重构
- `docs/superpowers/design-tokens.md` — 设计系统文档

**最低验收线：**
- 首页、座位列表、管理端三个核心页面截图对比，视觉一致性 > 90%
- 所有页面引用同一套 CSS Variables
- 导航栏有品牌标识区

### 升级项 2：统计页加入可视化图表

**当前问题表现：**
- `/Admin/Stats` 只有 4 个数字卡片（总数/今日预约/使用率/区域分布）
- 区域分布用文字列表而非图表，视觉上空白
- 无趋势数据，无时间维度对比

**为什么会拉低项目成熟度：**
统计页是管理端的"门面页面"，面试官通常会点进去看。数字文字列表是一个未完工的信号。

**对面试展示的价值：**
一个带柱状图/饼图的统计页，能立即证明你有前端图表库集成能力，这是招聘方看重的前端技能信号。

**对接近商业化的价值：**
BI 和数据可视化是几乎所有管理后台的标准功能。

**AI 可参与的工作点：**
- AI 推荐轻量级图表库（Chart.js 最适合本场景 — 30KB gzip，CDN 加载，无 build 工具依赖）
- AI 生成 Chart.js 初始化代码（柱状图/饼图/进度条）
- AI 提供 Controller 返回 JSON 数据的扩展点

**建议开发步骤：**
1. 在 `Admin/Stats.cshtml` 中从 CDN 加载 Chart.js（`https://cdn.jsdelivr.net/npm/chart.js`）
2. Controller 增加 2-3 个返回 JSON 的 Action（`GetAreaDistribution`, `GetHourlyTrend`, `GetDailyUsage`）
3. 前端用 Chart.js 渲染柱状图（区域分布）+ 饼图（状态分布）+ 折线图（7 日趋势）
4. 后端如果数据量不足，生成模拟历史数据填充图表（标注"模拟数据"）

**关键产物：**
- `AdminController` 新增 JSON API Action（2-3 个）
- `Views/Admin/Stats.cshtml` 增加 2-3 个 Canvas 图表区域
- Stats 视图模型扩展含图表数据

**最低验收线：**
- 统计页至少有 1 个柱状图 + 1 个饼图 + 1 个数字趋势
- 图表数据来自数据库（即使数据量少，图表框架正确渲染）
- 页面加载时图表有加载动画

### 升级项 3：友好错误页面体系（404/403/500）

**当前问题表现：**
- 访问不存在路径显示 ASP.NET Core 默认白底错误页
- 无管理员未授权友好提示
- 无全局异常处理页面

**为什么会拉低项目成熟度：**
白底黄页错误 = 最强烈的"未完工"信号。面试官如果演示时不小心输错 URL，看到的是框架默认错误页而非自定义页面，印象分会大幅降低。

**对面试展示的价值：**
自定义错误页面是"关注用户体验"的直接证据，面试中可以自然引出"我在项目中处理了全局异常和 404"。

**对接近商业化的价值：**
商业产品上线前必须自定义所有错误页面。

**AI 可参与的工作点：**
- AI 生成 404/403/500 页面 HTML + CSS（含友好文案和返回首页链接）
- AI 提供 Program.cs 中 `UseStatusCodePagesWithReExecute` 配置
- AI 提供全局异常过滤器（`IExceptionFilter` 或 `UseExceptionHandler`）

**建议开发步骤：**
1. 在 `Views/Shared/` 中创建 `Error.cshtml`（通用错误页，含错误消息和返回链接）
2. 在 `Views/Shared/` 中创建 `NotFound.cshtml`（404 专用页）
3. 在 `Views/Shared/` 中创建 `Forbidden.cshtml`（403 专用页，管理员未登录时跳转）
4. 修改 `Program.cs` — 配置 `UseStatusCodePagesWithReExecute`
5. 修改 `Program.cs` — 增强 `UseExceptionHandler` 指向自定义错误页

**关键产物：**
- `Views/Shared/Error.cshtml` — 通用错误页
- `Views/Shared/NotFound.cshtml` — 404 页面
- `Views/Shared/Forbidden.cshtml` — 403 页面
- `Program.cs` 错误处理中间件更新

**最低验收线：**
- 访问 `/nonexistent-page` → 显示自定义 404 页面
- 未登录访问 `/Admin/Reservations` → 显示自定义 403 或跳转登录页
- 页面风格与主站一致（引用相同 CSS）
- 所有错误页有返回首页/上一步的导航链接

### 升级项 4：全局 UI 交互打磨

**当前问题表现：**
- 导航栏无当前页高亮（问题 TD-06）
- 表单提交后无 loading 状态（用户可能重复点击）
- 操作成功/失败提示使用简单 alert 样式，无动画渐入渐出
- 表格行无 hover 高亮
- 无骨架屏或内容加载占位

**为什么会拉低项目成熟度：**
交互细节决定产品的"手感"。没有这些细节，用户（和面试官）会感觉操作"生硬"。

**对面试展示的价值：**
"我关注交互细节"是一个很好的面试回答角度，可以用具体代码证明。

**对接近商业化的价值：**
交互细节是沉默的用户体验投资，用户不会直接夸但离开时会察觉。

**AI 可参与的工作点：**
- AI 提供导航栏当前页高亮的多种实现方案
- AI 生成 CSS loading 动画（spinner / skeleton screen）
- AI 生成 toast 通知组件（成功/错误/警告三色 + 自动消失）
- AI 提供表单按钮防重复提交的 JS 片段

**建议开发步骤：**
1. **导航栏高亮** — 通过 ViewBag 或路由匹配给当前页 `<a>` 添加 `.active` 类 + CSS 样式
2. **表单 loading 状态** — 提交时按钮显示 spinner + disable，防止重复点击
3. **Toast 通知** — 在 _Layout.cshtml 中加入 Toast 容器，所有页面共享
4. **表格行 hover** — 如果已实现则确认，未实现则加 CSS `tr:hover`
5. **骨架屏** — 座位列表页和统计页加 CSS-only skeleton 占位

**关键产物：**
- `_Layout.cshtml` 更新（Toast 容器、导航高亮逻辑）
- `wwwroot/css/style.css` 增加交互相关样式
- `wwwroot/js/site.js`（新建，存放交互 JS）

**最低验收线：**
- 每个导航链接在对应页面高亮
- 表单提交按钮在请求中显示 spinner + 不可点击
- 操作成功后有 Toast 提示，3 秒自动消失
- 表格行 hover 背景变化

### 升级项 5：种子数据扩充 + 模拟数据生成器

**当前问题表现：**
- 仅 12 个种子座位（3 区域 × 4）
- 无预约种子数据（全部靠手动产生）
- 统计页因数据量少而展示单薄

**为什么会拉低项目成熟度：**
项目启动后如果数据太少，任何页面都显得空旷。面试官打开统计页看到"使用率 0%"，会质疑功能是否真正可用。

**对面试展示的价值：**
"我设计了种子数据生成器，确保演示时数据丰富" — 这展示了工程思维（考虑演示体验）。

**对接近商业化的价值：**
数据填充是所有 SaaS 产品 onboarding 的标准环节。

**AI 可参与的工作点：**
- AI 生成扩充后的座位数据（5 区域 × 8 座位 = 40 个）
- AI 生成模拟预约数据生成器（过去 30 天的预约记录）
- AI 提供种子数据版本控制策略

**建议开发步骤：**
1. 扩展 `DbInitializer.cs` — 座位从 12 扩充到 40（5 区域 × 8 座位，增加 4F/5F 区域）
2. 新增 `DbInitializer.GenerateSampleReservations()` — 生成最近 30 天的模拟预约数据
3. 确保模拟数据仅在开发环境生成（`app.Environment.IsDevelopment()` 守卫）
4. 为模拟数据标注"演示数据"字段或注释

**关键产物：**
- `Data/DbInitializer.cs` — 种子数据扩展
- 40 个座位（5 区域 × 8）+ 30 天模拟预约记录

**最低验收线：**
- 启动后数据库中有 ≥ 40 个座位
- 有过去 30 天的预约记录（≥ 50 条）
- 统计页数字卡片不再是 0
- 新增数据不影响已有自动化测试

### 升级项 6：部署可复现（Docker + dotnet publish 脚本）

**当前问题表现：**
- 无 `Dockerfile`
- 无 `docker-compose.yml`
- 无发布脚本（`publish.ps1` / `publish.sh`）
- 演示依赖本机 LocalDB + dotnet run

**为什么会拉低项目成熟度：**
"你跑一下看看" — 如果面试官想本地运行项目，需要装 SQL Server LocalDB + .NET 9 SDK，门槛较高。Docker 一键启动是专业信号。

**对面试展示的价值：**
Dockerfile + docker-compose 可以作为一个加分话题："我考虑了部署的可复现性"。

**对接近商业化的价值：**
Docker 容器化是几乎任何商业 ASP.NET Core 项目的标准交付方式。

**AI 可参与的工作点：**
- AI 生成多阶段构建 Dockerfile（build → publish → runtime）
- AI 生成 docker-compose.yml（含 SQL Server 容器 + app 容器）
- AI 生成发布脚本（`scripts/publish.ps1` 和 `scripts/deploy.sh`）

**建议开发步骤：**
1. 创建 `Dockerfile` — 多阶段构建（`mcr.microsoft.com/dotnet/sdk:9.0` build → `aspnet:9.0` runtime）
2. 创建 `docker-compose.yml` — app + SQL Server（或 SQLite 以减小体积）
3. 创建 `scripts/publish.ps1` — `dotnet publish` + 输出目录说明
4. 更新 README 添加 Docker 运行说明

**关键产物：**
- `Dockerfile`
- `docker-compose.yml`
- `scripts/publish.ps1`
- README 更新（Docker 运行节）

**最低验收线：**
- `docker-compose up` 后应用可在 `http://localhost:5207` 访问
- 数据库自动初始化（种子数据写入）
- 不用本机安装 .NET SDK 或 SQL Server

### 升级项 7：前端工程化（Vite + TypeScript 可选）

**当前问题表现：**
- 无任何前端 build 工具
- 使用 CDN 加载 Bootstrap 5.3 + Chart.js
- 无 TypeScript，JS 代码零散在 .cshtml 中
- 无 CSS 预处理

**为什么会拉低项目成熟度：**
面试官看到 CDN 引用的 Bootstrap + 内联 script 标签 = "未接触现代前端工具链"。

**对面试展示的价值：**
展示 Vite + TypeScript + PostCSS 集成到 ASP.NET Core 项目的能力，这在面试中是明确的技术信号。

**对接近商业化的价值：**
几乎所有商业 ASP.NET Core 项目都会使用前端工具链（至少是打包和类型检查）。

**AI 可参与的工作点：**
- AI 生成 Vite 配置（`vite.config.ts`）
- AI 生成 `tsconfig.json`
- AI 提供 ASP.NET Core + Vite 集成方案（开发时代理，生产时复制到 wwwroot）
- AI 将现有内联 JS 迁移到 TypeScript 模块

**建议开发步骤：**
1. 在项目根目录初始化 Vite + TypeScript
2. 将 `wwwroot/js/site.js` 迁移到 `TypeScript/src/main.ts`
3. 配置 Vite 输出到 `wwwroot/dist/`
4. 开发时用 Vite dev server 代理请求到 ASP.NET Core
5. _Layout.cshtml 引用打包后的资源而非 CDN

**关键产物：**
- `vite.config.ts`
- `tsconfig.json`
- `package.json`（更新，增加 Vite 依赖）
- `TypeScript/src/` 目录 + TypeScript 源码
- 更新后的 `_Layout.cshtml`

**最低验收线：**
- TypeScript 编译无错误
- 页面功能与之前完全一致
- 使用本地打包后的 Bootstrap + Chart.js 而非 CDN

## 4. 升级路线图

### 第一轮：展示级补强（建议 1-2 周，AI 可加速至 3-5 天）

**目标：** 消除"课堂作业"第一印象，让面试官愿意继续看下去。

**先后顺序：**

```
Step 1 ──── 友好错误页面体系（升级项 3）
            投入：0.5 天
            为什么先做：投入最小、见效最快、消除最强烈的"未完工"信号

Step 2 ──── 视觉设计系统重构（升级项 1）
            投入：1-2 天
            为什么第二：改变第一印象的核心战役

Step 3 ──── 全局 UI 交互打磨（升级项 4）
            投入：1 天
            为什么第三：在好看的基础上让操作有"手感"
```

## 5. 不建议现在就投入的过度设计项

| 项目 | 为什么不该现在做 |
|------|----------------|
| **微服务拆分** | 单一功能单体应用，微服务徒增复杂度，面试官也会觉得过度设计 |
| **Redis 缓存** | 数据量极小（12 座位），Session 使用内存缓存足够，加 Redis 无实际收益 |
| **SignalR 实时推送** | 座位状态变更不频繁，WebSocket 方案对面试加分有限且复杂度大增 |
| **OAuth 第三方登录** | 校园场景不需要，且增加安全暴露面 |
| **分库分表** | 数据量在百级别，完全用不上 |
| **Kubernetes 部署** | Docker Compose 足够演示 K8s 配置是过度工程 |
| **自动 CI/CD GitHub Actions** | 加分但非必要，且维护成本高。如 5 分钟能配好可做但不优先 |
| **i18n 多语言** | 校园场景无国际化需求 |
| **PWA 离线支持** | 座位预约是实时应用，离线无意义 |
| **单元测试全覆盖** | 当前 Playwright E2E 已覆盖主链路，先补 E2E 边界用例比补单元测试更高效 |

## 6. 最终结论

### 如果只继续打磨一轮，第一优先级应该做什么

**视觉设计系统重构 + 友好错误页面 + 统计页图表。**

这是投入产出比最高的组合：
- 3 项合计约 2-3 天（借助 AI）
- 直接消除"课堂作业"最大的 3 个信号
- 覆盖面试官最可能先看到的 3 个页面（首页、错误页、管理端统计页）

### 最能让面试官感受到项目成熟度的 3 个证据

| 证据 | 体现的能力 | 对应升级项 |
|------|-----------|-----------|
| **自定义 404/403/500 页面 + 全局异常处理** | 工程意识、用户体验关注 | 升级项 3 |
| **统计页 Chart.js 可视化（柱状图 + 饼图）** | 前端集成能力、数据展示意识 | 升级项 2 |
| **设计 Token 体系（CSS Variables 驱动的主题系统）** | 前端工程化思维、代码组织能力 | 升级项 1 |

这三个证据加在一起，能在面试官打开仓库的前 60 秒内建立"这个学生有真实项目经验"的印象。
