## 1. 数据设计目标

| 目标 | 说明 |

| 支撑 9 页面全部数据需求 | 用户首页、座位列表、座位详情、预约提交、我的预约、管理员登录、预约管理、座位管理、统计 |

## 3. 关系说明

### ER 图（文字描述）

```
StudentUser (1) ────< (N) Reservation (N) >──── (1) Seat
```

- 一个 **StudentUser** 可以有 **多条 Reservation**
- 一个 **Seat** 可以有 **多条 Reservation**
- **Reservation** 通过 `StudentUserId` 关联 StudentUser，通过 `SeatId` 关联 Seat

### 外键关系

| 子表 | 外键字段 | 父表 | 删除行为 |
|------|---------|------|---------|
| `Reservations` | `SeatId` | `Seats.Id` | CASCADE（删除座位时同时删除其预约记录） |
| `Reservations` | `StudentUserId` | `StudentUsers.Id` | CASCADE（删除学生时同时删除其预约记录） |

### 4.3 Reservations（预约记录）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `Id` | `int` | PK, 自增 | 主键 |
| `SeatId` | `int` | FK → Seats.Id, NOT NULL | 外键：被预约的座位 |
| `StudentUserId` | `int` | FK → StudentUsers.Id, NOT NULL | 外键：预约人 |
| `Date` | `date` | NOT NULL | 预约日期（服务器当天，不含时间部分） |
| `StartTime` | `time` | NOT NULL | 预约开始时间，如 `14:00:00` |
| `EndTime` | `time` | NOT NULL | 预约结束时间，如 `16:00:00` |
| `Status` | `nvarchar(20)` | NOT NULL, DEFAULT 'Pending' | 持久化状态：`Pending` / `Cancelled` |
| `CreatedAt` | `datetime2` | NOT NULL, DEFAULT GETDATE() | 预约创建时间 |

**DDL：**

```sql
CREATE TABLE Reservations (
    Id INT IDENTITY(1,1) PRIMARY KEY,
    SeatId INT NOT NULL,
    StudentUserId INT NOT NULL,
    Date DATE NOT NULL,
    StartTime TIME NOT NULL,
    EndTime TIME NOT NULL,
    Status NVARCHAR(20) NOT NULL DEFAULT 'Pending',
    CreatedAt DATETIME2 NOT NULL DEFAULT GETDATE(),
    CONSTRAINT FK_Reservations_Seats FOREIGN KEY (SeatId) REFERENCES Seats(Id) ON DELETE CASCADE,
    CONSTRAINT FK_Reservations_StudentUsers FOREIGN KEY (StudentUserId) REFERENCES StudentUsers(Id) ON DELETE CASCADE
);
```

## 5. 关键字段说明

### 5.1 为什么 `Date` 用 `date` 类型，而 `StartTime`/`EndTime` 用 `time` 类型？

分开存储而不是用单个 `DateTime`，原因：

| 方案 | 问题 |
|------|------|
| 一个 `DateTime` 字段存开始，一个存结束 | 跨天查询需要复杂的范围条件 |
| `Date` + `StartTime`/`EndTime` 分开 | 按日期查询直接 `WHERE Date = @today`，按时段查询直接比较 `time` |

**查询示例：** 查询某座位今日某时段是否被预约

```sql
SELECT COUNT(*) FROM Reservations
WHERE SeatId = @seatId
  AND Date = @today
  AND Status = 'Pending'
  AND StartTime < @endTime
  AND EndTime > @startTime;
```

### 5.2 `Status` 为什么用 `nvarchar` 而不是 `int` enum？

| 理由 | 说明 |
|------|------|
| 可读性 | 数据库直接看到 `'Pending'` 而不是 `0`，调试方便 |
| 扩展性 | 新增状态不需要改代码枚举定义 |
| 学生友好 | 不引入 EF Core 枚举转换的复杂度 |

## 6. 状态字段设计

### 6.1 座位状态（Seat.Status）

| 数据库值 | 显示值 | 说明 |
|---------|--------|------|
| `Available` | 空闲 | 可被预约 |
| `Maintenance` | 维护中 | 管理员标记，不可预约 |

**不存库的状态（动态计算）：**

| 显示值 | 计算逻辑 |
|--------|---------|
| 已预约 | 该座位今天有 `Pending` 状态的预约且 `StartTime > 当前时间` |
| 使用中 | 该座位今天有 `Pending` 状态的预约且 `StartTime ≤ 当前时间 < EndTime` |

### 6.2 预约状态（Reservation.Status）

| 数据库值 | 显示值 | 说明 |
|---------|--------|------|
| `Pending` | 待开始 | 预约成功，等待生效 |
| `Cancelled` | 已取消 | 用户手动取消 |

**不存库的状态（动态计算）：**

| 显示值 | 计算逻辑 |
|--------|---------|
| 使用中 | `StartTime ≤ 当前时间 < EndTime` 且 Status = `Pending` |
| 已完成 | `当前时间 ≥ EndTime` 且 Status = `Pending` |

### 6.3 状态流转汇总

| 实体 | DB 持久化 | 动态计算 | 计算依据 |
|------|----------|---------|---------|
| 座位 | Available, Maintenance | 已预约, 使用中 | 关联的 Reservation 记录 + 当前时间 |
| 预约 | Pending, Cancelled | 使用中, 已完成 | 自身的 StartTime/EndTime + 当前时间 |

## 7. 查询与筛选字段设计

### 7.1 索引设计

```sql
-- 索引 1：座位列表按区域/楼层筛选
CREATE INDEX IX_Seats_Area_Floor ON Seats (Area, Floor);

-- 索引 2：预约冲突检查（同一座位同一日期的有效预约）
CREATE INDEX IX_Reservations_SeatId_Date ON Reservations (SeatId, Date) INCLUDE (StartTime, EndTime, Status);

-- 索引 3：我的预约查询（按用户）
CREATE INDEX IX_Reservations_StudentUserId ON Reservations (StudentUserId) INCLUDE (Date, StartTime, EndTime, Status);

-- 索引 4：管理端预约管理按时间倒序
CREATE INDEX IX_Reservations_Date_StartTime ON Reservations (Date DESC, StartTime DESC);
```

### 7.2 主要查询场景与对应 SQL

| 场景 | 查询方式 | 涉及表 |
|------|---------|--------|
| 座位列表（全部） | `SELECT * FROM Seats ORDER BY SeatNumber` | Seats |
| 座位列表（筛选） | `SELECT * FROM Seats WHERE Area = @area AND Floor = @floor` | Seats |
| 座位详情 | `SELECT * FROM Seats WHERE Id = @id` | Seats |
| 今日预约时段 | `SELECT * FROM Reservations WHERE SeatId = @id AND Date = @today ORDER BY StartTime` | Reservations + StudentUsers |
| 冲突检查 | `SELECT COUNT(*) FROM Reservations WHERE SeatId = @id AND Date = @today AND Status = 'Pending' AND StartTime < @end AND EndTime > @start` | Reservations |
| 我的预约 | `SELECT * FROM Reservations WHERE StudentUserId = @uid ORDER BY Date DESC, StartTime DESC` | Reservations + Seats |
| 管理端全部预约 | `SELECT * FROM Reservations ORDER BY Date DESC, StartTime DESC` | Reservations + Seats + StudentUsers |
| 管理端座位列表 | `SELECT * FROM Seats ORDER BY SeatNumber` | Seats |
| 动态座位状态 | `SELECT SeatId, StartTime, EndTime FROM Reservations WHERE Date = @today AND Status = 'Pending'` | Reservations |

### 7.3 筛选字段说明

| 筛选维度 | 对应字段 | 使用方式 |
|---------|---------|---------|
| 区域筛选 | `Seat.Area` | URL 参数 `?area=一楼大厅` |
| 楼层筛选 | `Seat.Floor` | URL 参数 `?floor=1F` |
| 日期筛选（管理端） | `Reservation.Date` | URL 参数 `?date=2026-07-13` |
| 状态筛选（管理端） | `Reservation.Status` + 动态状态 | 代码中先按 Status 过滤再计算动态状态 |
| 用户筛选（管理端） | `Reservation.StudentUserId` | URL 参数 `?userId=1` |

## 8. 统计字段与统计口径

### 8.1 统计指标

| 指标 | 计算方式 | 使用场景 |
|------|---------|---------|
| 座位总数 | `SELECT COUNT(*) FROM Seats` | 统计页 |
| 今日预约数 | `SELECT COUNT(*) FROM Reservations WHERE Date = @today` | 统计页 |
| 座位使用率 | `(今日有预约的座位数 / 座位总数) × 100%` | 统计页 |
| 各区域座位分布 | `SELECT Area, COUNT(*) FROM Seats GROUP BY Area` | 统计页 |

### 8.2 使用率计算细节

```sql
-- 今日有预约的座位数（去重）
SELECT COUNT(DISTINCT SeatId) 
FROM Reservations 
WHERE Date = @today AND Status = 'Pending';

-- 代码中计算
var usageRate = reservedSeatCount > 0 
    ? (double)distinctSeatCount / totalSeatCount * 100 
    : 0;
```

### 8.3 统计说明

- 所有统计数字**实时计算**，不建统计汇总表
- 使用率的分母是座位总数（含维护中座位），分子是今日有 Pending 预约的座位数（去重）
- 本期统计不需要历史趋势图，不需要时间维度聚合

## 9. 数据一致性与约束说明

### 9.1 业务约束

| 约束 | 实现方式 | 违反后果 |
|------|---------|---------|
| 同一座位同一日期同一时段不可重复预约 | 代码中 `ReservationController.Create(POST)` 执行冲突查询 | 提示"该时段已被预约" |
| 最短预约 30 分钟 | 代码中 `(endTime - startTime).TotalMinutes >= 30` | 提示"最短预约时长为 30 分钟" |
| 最长预约 4 小时 | 代码中 `(endTime - startTime).TotalHours <= 4` | 提示"最长预约时长为 4 小时" |
| `EndTime` 必须晚于 `StartTime` | 代码中比较 | 提示"结束时间必须晚于开始时间" |
| 仅"待开始"可取消 | 代码中检查 `reservation.Status == "Pending"` | 提示"只能取消待开始的预约" |
| 只能取消自己的预约 | 代码中检查 `reservation.StudentUserId == Session.UserId` | 提示"只能取消自己的预约" |

### 9.2 数据库层约束

| 约束类型 | 说明 |
|---------|------|
| 外键约束 | `Reservations.SeatId` → `Seats.Id`，`Reservations.StudentUserId` → `StudentUsers.Id` |
| 唯一约束 | 暂无（座位编号不设唯一约束，允许未来有相同编号不同区域的设计） |
| CHECK 约束 | 不使用数据库 CHECK，全部由应用层代码校验 |
| NULL 约束 | 所有业务字段均为 NOT NULL |

> 为什么不在数据库层加预约冲突 CHECK 约束？
> EF Core 的 `EnsureCreated` 和迁移工具对 CHECK 约束支持不友好。本项目的并发量极低（课堂演示），应用层锁 + 数据库事务足以保证一致性。不需要数据库层的复杂约束。

### 9.3 并发场景说明

本项目是课堂演示项目，不考虑高并发。最可能出现的并发场景是"两个学生同时预约同一个座位的同一时段"。处理方式：

```csharp
// 在同一个 DbContext 实例中，先查询再插入
var conflict = _db.Reservations.Any(r => /* 冲突条件 */);
if (!conflict)
{
    _db.Reservations.Add(new Reservation { ... });
    _db.SaveChanges(); // SaveChanges 是原子操作
}
```

如果两个请求同时到达同一台服务器的同一个进程内，EF Core 的 `SaveChanges` 在 SQL Server 层面是事务性的，第二个请求会因为查询到冲突而失败。这个级别对课堂项目足够了。

## 11. 默认补足项 / 当前假设

### 补足项

前序文档已给出完整的数据骨架（PRD-Lite 第 11 节 × 页面树第 6 节 × 系统设计说明第 3 节），本次未引入默认起点中的额外实体。

以下为前序文档未明确、本次补充的细节：

| 补足项 | 本次决策 | 依据 |
|--------|---------|------|
| `Date` 字段类型 | `date`（不是 `datetime`） | 按日期查询需求明确，分开 `date` + `time` 比 `datetime` 更清晰 |
| 索引设计 | 4 个覆盖索引 | 从查询场景反推，保证冲突检查和排序性能 |
| 并发处理方式 | EF Core SaveChanges 事务保障 | 课堂项目，不考虑高并发 |
| `SeatNumber` 唯一性 | 不设 UNIQUE 约束 | 允许未来不同区域有相同编号 |
| `Admin` 是否建表 | **不建表** | 系统设计说明第 10 节明确"管理员认证使用硬编码 + Session" |

### 当前假设

| 假设 | 影响 |
|------|------|
| 每次预约只涉及**一个**座位（不是批量预约） | Reservation 表设计为单座位粒度 |
| 预约不跨天 | `Date` 字段无时间部分，查询简化 |
| 同一用户的预约在同一时段**可以**重叠（约不同座位） | 冲突检查只检查同一 SeatId |
| 系统运行在开发模式（不处理生产级异常） | 无数据库重试逻辑、无死锁处理 |
