## C. 必改项

### 必改 1：技术栈版本统一

**问题**：立项单写 .NET 8，但环境装的是 .NET 9。

**修改建议（二选一）**：
- 方案 A：将技术栈改为 `.NET 9`（就地使用，省去额外安装步骤）
- 方案 B：保留 `.NET 8`，但在立项单中增加"前置条件：安装 .NET 8 SDK"的说明

推荐方案 A 走 .NET 9，因为已经在用。

### 必改 2：成功标准 #9 过于绝对

**问题**："项目能通过 `dotnet run` 一键启动，无需额外配置数据库连接"——LocalDB 需要：
1. 本机安装了 SQL Server LocalDB
2. `appsettings.json` 里有正确的连接字符串
3. 首次运行需要执行 `dotnet ef database update`

**修改方向**：改为"按项目 README 完成 LocalDB 配置后，通过 `dotnet run` 可正常启动"。

### 必改 3：取消预约不是独立页面

**问题**：取消预约的实际流程是"在我的预约页点击取消 → 确认 → 状态变更 → 跳回列表"，不需要一个独立的 .cshtml 页面。

**修改方向**：从"页面分布"中移除 `/Reservation/Cancel/{id}` 独立路由，改为在"我的预约"页功能描述中注明"提供取消操作按钮"。或者在说明栏注明"此路由为 POST 操作，不单独渲染页面"。

## D. 建议优化项

### 建议 1：明确模拟登录方式

建议说明"轻量模拟登录"的具体交互方式，例如：
> 在用户首页顶部显示一个下拉框，列出预设的体验账号（学生A~学生E），选中后即登录。用户信息存储在 Session 中，刷新页面后保持登录状态。

给出明确方案，后续设计页面时就不会出现"登录页面要不要做""路由怎么设计"的纠结。

### 建议 2：统计页的技术选型建议

如果学生不熟悉前端图表库，统计页做图表可能卡住。建议明确：

- 用 **Chart.js**（最轻量，引入一个 CDN 即可）
- 或者直接用 **HTML 表格 + 百分比数字**（不会画图也能做）
- 本期不做柱状图/饼图也不影响验收

建议在"本期必须完成内容"的统计页一项中注明使用的方案。

### 建议 3：座位详情页空状态

座位详情页需要说明"该座位今日无预约"时的界面表现。例如：

> 如果当前日期没有预约记录，显示"今日暂无预约记录，该座位当前空闲"。

### 建议 4：里程碑 1 拆细

对动手能力一般的同学，第 1 天从零到"种子数据入库"压力较大。建议：

- 把里程碑 1 拆成"1a: 项目创建 + 模型设计"和"1b: 数据库 + 种子数据"
- 或者在第 1 天只要求"能通过编译"，"种子数据入库"放入第 2 天上午

## E. 是否可以进入下一步

**可以，但需先落实必改项 1（技术栈版本）。** 必改项 2 和 3 可以在进入"需求分析与 MVP 确认"阶段时同步修正。

理由是：
- 立项单的框架、范围、页面分布、成功标准已满足了进入下一阶段的底线要求。
- "技术栈版本"会影响下一步创建项目时的具体命令（`dotnet new mvc`），所以需要先确认。
- 其他问题可以在后续阶段逐步打磨，不阻塞进度。
