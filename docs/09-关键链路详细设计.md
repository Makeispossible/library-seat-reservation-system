# 关键链路详细设计 — 图书馆座位预约系统

> 本文档基于 `docs/04-页面树与业务流程.md`、`docs/07-系统设计说明.md`、`docs/08-数据库设计.md` 编写。

---

## 1. 关键链路选择说明

前序文档已明确 5 条链路，本次选取其中 **3 条**进行详细展开：

| 链路 | 名称 | 优先级 | 选择原因 |
|------|------|--------|---------|
| **链路 A** | 完整预约流程（5 步闭环） | 核心链路 | 覆盖用户首页 → 座位浏览 → 预约提交 → 结果查看的完整流程，涉及最多页面和动作 |
| **链路 B** | 取消预约流程（3 步闭环） | 核心链路 | 涉及状态变更、权限校验、前端弹窗交互，是第二高频操作 |
| **链路 D** | 管理员登录 + 管理座位（4 步闭环） | 管理链路 | 覆盖管理端认证 → 查看 → 写入的完整管理流程，涉及 Session 守卫 |

**链路 C（多账号对比）** 和 **链路 E（查看统计）** 不展开详细设计：
- 链路 C 是链路 A 的验证性变体，不引入新代码路径
- 链路 E 是纯查询展示（无写入、无状态变更），在数据库设计（第 8 节）中已覆盖统计口径，Controller 和 View 代码直出即可

## 2. 链路 A：完整预约流程

### 2.1 页面与路由

```
步骤 | 页面 | 路由 | HTTP 方法
①    | 用户首页       | GET  /                                              | GET
②    | 座位列表页     | GET  /Seats[?area=xxx&floor=xxx]                    | GET
③    | 座位详情页     | GET  /Seats/Detail/{id}                              | GET
④    | 预约提交页     | GET  /Reservation/Create/{id}                       | GET
```

**路由参数说明：**
- ③ 的 `{id}` → 座位 Id
- ④ 的 `{id}` → 座位 Id（因审计建议，参数名从 `seatId` 改为 `id` 以匹配默认路由）

#### 步骤 ③ → SeatsController.Detail(int id)

```csharp
// 职责：

//   2. 计算座位的当前显示状态

    .OrderBy(r => r.StartTime)

    seat.Status,
    todayReservations.Where(r => r.Status == "Pending").ToList());
```

**异常/边界：**

| 场景 | 处理 |
|------|------|
| 座位 Id 不存在 | `return NotFound()` → 404 |
| 该座位今日无预约 | 视图显示"今日暂无预约记录" |
| 座位非空闲 | "预约此座位"按钮不显示，显示"当前不可预约" |
| 座位维护中 | 显示"维护中"标识，按钮隐藏 |

#### 步骤 ④ → ReservationController.Create(GET)

```csharp
// GET：渲染预约表单
// 前置检查：
//   1. Session 中有 UserId？→ 无则 TempData 错误 + 重定向首页
//   2. 座位当前是否空闲？→ 非空闲则 TempData 错误 + 重定向详情页
// ViewBag.Seat = 当前 Seat 实体
```

#### 步骤 ④ → ReservationController.Create(POST)

```csharp
// POST：校验 + 写入 + 重定向
// 接收参数：int id (seatId), TimeSpan startTime, TimeSpan endTime

// 校验顺序（任一失败 → 返回表单 + 错误消息）：
//   1. Session 中 UserId 是否存在
//   2. endTime > startTime
//   3. (endTime - startTime).TotalMinutes >= 30
//   4. (endTime - startTime).TotalHours <= 4
//   5. 冲突检查（同一座位同一日期同一时段不可重复）

// 冲突检查 SQL 等价逻辑：
var conflict = _db.Reservations.Any(r =>
    r.SeatId == id &&
    r.Date == today &&
    r.Status == "Pending" &&
    r.StartTime < endTime &&
    r.EndTime > startTime);

// 写入：
_db.Reservations.Add(new Reservation
{
    SeatId = id,
    StudentUserId = userId.Value,
    Date = today,
    StartTime = startTime,
    EndTime = endTime,
    Status = "Pending",
    CreatedAt = DateTime.Now
});
_db.SaveChanges();

// 成功后重定向
TempData["SuccessMessage"] = "预约成功！";
return RedirectToAction("My");
```

**异常/边界（完整清单）：**

| # | 场景 | 检测时机 | 处理方式 |
|---|------|---------|---------|
| 1 | 未选择体验账号 | GET 入口 | 重定向首页 + TempData 错误 |
| 2 | 座位不存在 | GET 入口 | return NotFound() |
| 3 | 座位不空闲已变更 | POST 提交时 | 返回表单 + 错误"该座位当前不可预约" |
| 4 | 结束时间 ≤ 开始时间 | POST 校验 | 表单 + 错误"结束时间必须晚于开始时间" |
| 5 | 不足 30 分钟 | POST 校验 | 表单 + 错误"最短预约时长为 30 分钟" |
| 6 | 超过 4 小时 | POST 校验 | 表单 + 错误"最长预约时长为 4 小时" |
| 7 | 时段冲突 | POST 冲突查询 | 表单 + 错误"该时段已被预约" |
| 8 | 数据库写入失败 | SaveChanges | ASP.NET Core 默认错误页 |

#### 步骤 ⑤ → ReservationController.My()

```csharp
// 职责：
//   1. 检查 Session 中 UserId
//   2. 查询当前用户的所有 Reservation（按时间倒序）
//   3. 为每条记录计算动态状态
//   4. 标记可取消的预约（Pending 且 startTime > 当前时间）

var reservations = _db.Reservations
    .Include(r => r.Seat)
    .Where(r => r.StudentUserId == userId)
    .OrderByDescending(r => r.Date)
    .ThenByDescending(r => r.StartTime)
    .ToList();

ViewBag.Reservations = reservations.Select(r => new
{
    r.Id,
    SeatNumber = r.Seat.SeatNumber,
    r.Seat.Area,
    r.Date,
    r.StartTime,
    r.EndTime,
    DisplayStatus = StatusHelper.GetReservationDisplayStatus(r.Status, r.StartTime, r.EndTime),
    CanCancel = r.Status == "Pending" && DateTime.Now.TimeOfDay < r.StartTime
}).ToList();
```

**异常/边界：**

| 场景 | 处理 |
|------|------|
| 未登录 | 重定向首页 + TempData 错误 |
| 该用户无预约记录 | 视图显示空状态"你还没有预约记录" + "去预约"按钮 |
| 有记录但全是已取消/已完成 | 正常显示，无取消按钮 |

### 2.3 Service 职责

本项目**不引入独立 Service 层**。链路 A 中的业务逻辑分布在：

| 逻辑 | 位置 | 说明 |
|------|------|------|
| 动态状态计算 | `Helpers/StatusHelper.cs` | 两个静态方法，纯计算无副作用 |
| 冲突校验 | `ReservationController.Create(POST)` | Controller 中直接 LINQ 查询 |
| 时间规则校验 | `ReservationController.Create(POST)` | Controller 中直接判断 |
| 权限校验（体验账号） | `ReservationController.Create(GET/POST)` | 检查 Session UserId |

### 2.6 状态流转

链路 A 不改变任何持久化状态，仅在视图层计算动态状态。状态变化发生时刻：

| 时刻 | 座位显示状态变化 | 预约记录显示状态变化 | 备注 |
|------|-----------------|---------------------|------|
| 预约成功后 | 空闲 → 已预约（动态） | —（新增 Pending） | 数据库新增 Pending 记录 |
| 当前时间 ≥ StartTime | 已预约 → 使用中（动态） | 待开始 → 使用中（动态） | 不写库 |
| 当前时间 ≥ EndTime | 使用中 → 空闲（动态） | 使用中 → 已完成（动态） | 不写库 |

### 2.7 异常与边界情况（汇总）

| 类别 | 场景 | 处理 |
|------|------|------|
| 权限 | 未选体验账号访问预约页 | 重定向首页 |
| 权限 | 未选体验账号访问我的预约 | 重定向首页 |
| 数据不存在 | 座位 Id 无效 | 404 |
| 业务冲突 | 座位已被其他人预约相同时间段 | 表单错误提示 |
| 业务冲突 | 提交瞬间座位被管理员设为维护中 | POST 时二次校验 |
| 时间规则 | 选时不足 30 分钟或超过 4 小时 | 表单错误提示 |
| 时间规则 | 开始时间 ≥ 结束时间 | 表单错误提示 |
| 空数据 | 无座位 | 空状态显示 |
| 空数据 | 无预约记录 | 空状态显示 |
| 前端 | 用户刷新提交页（重复提交） | PRG 模式（POST → Redirect → GET） |

## 3. 链路 B：取消预约流程

### 3.1 页面与路由

```
步骤 | 页面    | 路由                          | HTTP 方法
①    | 我的预约 | GET  /Reservation/My          | GET
②    | 弹窗确认 | （Bootstrap Modal，纯前端）   | — 
③    | 执行取消 | POST /Reservation/Cancel/{id} | POST
     | 结果页   | （重定向到我的预约页）        | —
```

### 3.2 Controller 动作

```csharp
// ReservationController.Cancel(int id) [HttpPost]
var reservation = _db.Reservations.Include(r => r.Seat)
    .FirstOrDefault(r => r.Id == id);

// 边界 1：预约不存在
if (reservation == null) return NotFound();

// 边界 2：不是待开始状态
if (reservation.Status != "Pending")
{
    TempData["ErrorMessage"] = "只能取消待开始的预约";
    return RedirectToAction("My");
}

// 边界 3：不是本人的预约
if (reservation.StudentUserId != userId)
{
    TempData["ErrorMessage"] = "只能取消自己的预约";
    return RedirectToAction("My");
}

// 边界 4：已经开始
if (DateTime.Now.TimeOfDay >= reservation.StartTime)
{
    TempData["ErrorMessage"] = "已开始的预约无法取消";
    return RedirectToAction("My");
}

// 执行取消
reservation.Status = "Cancelled";
_db.SaveChanges();

TempData["SuccessMessage"] = "预约已取消";
return RedirectToAction("My");
```

### 3.3 状态流转

```
操作前：Reservation.Status = "Pending"（DB）
操作后：Reservation.Status = "Cancelled"（DB）

座位动态状态变化：已预约 → 空闲（下次查询时自动计算）
```

### 3.4 异常与边界情况

| 场景 | 检测方式 | 处理 |
|------|---------|------|
| 预约不存在 | `FirstOrDefault` → null | 404 |
| 预约已取消 | `Status != "Pending"` | TempData 错误 + 重定向 |
| 预约已完成 | `Status != "Pending"` + 动态判断 | 同上 |
| 预约正使用中 | `Status == "Pending"` 但时间已到 | 通过 `timeOfDay >= StartTime` 拦截 |
| 取消别人的预约 | `StudentUserId != Session.UserId` | TempData 错误 + 重定向 |
| 数据库写入失败 | `SaveChanges` | ASP.NET Core 默认错误页 |
| 重复点击取消按钮 | 前端 Modal 防止 + POST 幂等 | 第二次 POST 因 Status 已非 Pending 而失败 |

### 4.3 [AdminAuth] 守卫机制

```csharp
public class AdminAuthAttribute : ActionFilterAttribute
{
    public override void OnActionExecuting(ActionExecutingContext context)
    {
        if (context.HttpContext.Session.GetString("AdminLoggedIn") != "true")
        {
            context.Result = new RedirectToActionResult("Login", "Admin", null);
        }
    }
}
```

应用范围：

| Action | 守卫 |
|--------|------|
| `AdminController.Login()` | ❌ 不守卫（登录页本身不需要登录） |
| `AdminController.Logout()` | ❌ 不守卫（登出操作无需登录） |
| `AdminController.Reservations()` | ✅ [AdminAuth] |
| `AdminController.Seats()` | ✅ [AdminAuth] |
| `AdminController.CreateSeat()` | ✅ [AdminAuth] |
| `AdminController.Stats()` | ✅ [AdminAuth] |

| 操作 | 数据库变化 | 影响 |
|------|-----------|------|
| 管理员登录 | 无（Session 写入） | — |
| 管理员登出 | 无（Session 清除） | — |

## 6. 默认补足链路 / 当前假设

前序文档已明确链路 A/B/D/E 的定义，本次未引入默认链路（如"支付"、"签到"等）。以下为前序文档未详细写清、本次补充的设计细节：

| 补足项 | 本次决策 | 依据 |
|--------|---------|------|
| 取消预约的"已开始"边界 | `timeOfDay >= StartTime` 才不允许取消 | 防止用户在开始时间刚过一秒后还能取消 |
| 迟到/未签到处理 | 不做。预约开始后自动变为"使用中"，结束后变为"已完成" | 课堂项目，不做超时释放逻辑 |
| 管理端多维度查询 | 本期不做按日期/状态/用户筛选，仅展示全部 | 系统设计第 7 节虽列出了筛选维度，但降低复杂度优先 |
| 预约冲突的精确区间 | `StartTime < endTime AND EndTime > startTime` | 标准区间重叠判断，覆盖所有重叠情况 |
| 添加座位的唯一性 | 精确匹配 `SeatNumber`（字符串完全相等） | 不区分大小写（数据库默认不区分） |
