## 3. 做得好的地方

### 架构与设计
- **体验账号切换**：无需注册即可使用，降低演示门槛
- **静态动态双状态设计**：Seat.Status 持久化仅 Available/Maintenance，前端通过 StatusHelper 动态计算"使用中/待开始/已完成/已预约"等状态，避免状态爆炸
- **[AdminAuth] 过滤器**：管理端 Session 守卫以过滤器统一控制，无需每个 Action 重复检查

### 工程实践
- **文档-原型-代码证据链**：每个阶段的文档、原型和代码可互相验证
- **审计跟踪**：每个阶段都有审计文档，记录与前一阶段的偏差和修正
- **循序渐进**：从 Sprint 0 骨架到 Sprint 4 联调测试，每轮明确最低完成线

### 测试
- **双重烟雾测试**：Playwright 自动点击 + Node.js 脚本 fetch，互为容灾
- **不依赖 Playwright 浏览器下载**：使用系统 msedge，减少 CI 配置复杂度

## 6. 最终演示顺序建议

以下演示顺序约 5-8 分钟，覆盖全部主要功能：

```
Step 1: 启动应用（30 秒）
  dotnet run --project src/LibrarySeatReservation.Web

Step 2: 用户端主链路（3 分钟）
  ① 首页 → 切换学生A（导航栏显示"学生A"）
  ② 座位列表 → 按区域筛选 → 点击 A-01 进入详情
  ③ 选择时段 10:00-11:00 → 提交预约 → 跳转我的预约
  ④ 查看预约记录 → 取消预约 → 确认 → 成功提示

Step 3: 管理端主链路（3 分钟）
  ⑤ 访问 /Admin/Login → 输入 admin/admin123
  ⑥ 预约管理 → 按状态筛选 → 管理员取消预约
  ⑦ 座位管理 → 切换座位状态 Available ↔ Maintenance
  ⑧ 统计页 → 查看总数/今日预约/使用率/区域分布
  ⑨ 退出登录 → 跳转登录页

Step 4: 自动化测试（30 秒展示，可在 Step 1 前并行启动）
  node scripts/smoke-test.mjs        # 10/10 通过
  npx playwright test --reporter=list # 9/9 通过
```
